## 0. 30 秒摘要

**核心決策:事件理解卡為全班標配,三頁式報告轉為選配延伸。**

- 全班必交:事件理解卡(三句展示稿為核心)

- 觀課敘事主軸:**「學生有判斷,不是貼 AI」** ─ 由「我改掉哪個立場詞 / 為什麼改」這格承擔

## 1. 課堂版本決策(最終版)

| 項目 | 決策 | 理由 |
|---|---|---|
| 全班標配 | 事件理解卡 | 跑班 / 動機差距大,理解卡保住最低學習產出 |
| 選配延伸 | 三頁式報告 | 不懲罰低動機學生,給高成就學生深挖空間;作為展示加值,不作基本分門檻 |
| 評分主體 | `student_card.schema.json` | 延伸報告只加分,不影響基本通過 |
| 觀課展示 | 理解卡為主,延伸報告為輔 | 觀課者最敏感的是「學生有判斷」,不是篇幅 |

商二乙觀課班與跑班學生使用**同一份理解卡**,不再分裂兩套。

## 2. 事件理解卡 — 設計細節

### 2.1 九大項保留,以下三處針對低動機降門檻

**(1) 立場詞詞卡(嵌在「我改掉 AI 一個地方」這格的側欄)**

下表為合成版,加入 Codex 從 PDF 原典補回的日治詞彙。「過度簡化風險」欄提醒老師批改時若學生用得太武斷,可拉回討論。

| 詞彙 | 漢/日視角 | 族視角 | 適用事件 | 過度簡化風險 |
|---|---|---|---|---|
| 招撫 | 招撫原住民、撫番 | 國家力量介入、要求服從 | 大港口 | 不宜一律譯成「強迫歸順」 |

| 滅社 | 喪身滅社、討伐滅社 | 部落被拆散、遷徙、失去故土 | 大港口、七腳川 | 「清軍屠殺」只適大港口;七腳川是日治「討伐滅社」 |

**(2) 同儕互評改成「一勾一改」(替代原模板的開放式建議欄)**

- 勾選一個:[ ] 這句寫得清楚 [ ] 不夠具體 [ ] 我看不懂
- 只改一個句子(不要求完整評論)

理由:高中同儕互評的重點不是評鑑能力,是讓作者拿到「可以立刻修改的最小回饋」。

**(3) 不另設「為何被跳過」這類提示**

(原版設計曾建議對七腳川另設提示。**已撤回** ─ 所有四事件在卡片中地位平等。)

### 2.2 卡片視覺設計

- **正面(無互動)**:三句展示稿粗體大字
- **下半(可展開 details)**:AI 提問與重點 / 原句 vs 改寫 / 觀點對照 / 來源
- **不做 CSS 翻面動畫** — 對快速批改 / 觀課掃看是過度設計
- **AI 原句 → 學生改寫的視覺**:刪除線標原句立場詞 + 對比色高亮替換詞

### 2.3 評分軌道嵌在卡片右側

四項評量(0-2 分)以細條形式直接放在卡片右側,Lai 在電腦教室同頁勾分,不必切頁:

```
┌─ 學生卡片內容 ─┐  ┌ 評分 ┐
│  三句展示稿     │  │ 基本事實 □□ │
│  AI 過程       │  │ AI 使用  □□ │
│  觀點對照      │  │ 多視角   □□ │
│  ...          │  │ 三句展示 □□ │
└────────────────┘  └ 8/8 ┘
```

> **不另開 `grading_sheet.html`**:那適合期末彙整,不適合即時逐張快評(Codex 立場,Claude 同意)。

## 5. 資料結構增補

### 5.1 新增 `student_card.schema.json`

不污染現有 `class_data.schema.json`(供延伸三頁式繼續用)。共用 `event` 列舉與 `sources` 結構。

### 5.2 sources 增加 pdf 引用欄

```json
{
  "type": "NotebookLM",
  "title": "原民會叢書-大港口事件",
  "url": "",
  "pdf_page": "P. 23 第 3 段"
}
```

理由:NotebookLM 答案的可稽核性是專案第一原則(正確度優先於美觀)。沒這欄,課後查核很難。

### 5.3 卡片結構樹

```
event_card
├─ identity (class_name, seat, name, event)
├─ facts (time, place, ethnic, people)
├─ ai_process
│  ├─ question
│  ├─ ai_points (3)
│  ├─ keywords (3)
│  ├─ original_ai_sentence
│  ├─ revised_sentence
│  └─ revision_reason
├─ perspectives (official, tribal, difference)
├─ landscape (types[], looks_like_now, after_knowing_history)
├─ showcase_sentences (fact, difference, reflection) ← 卡片正面 / 地圖素材
├─ sources (含 pdf_page)
├─ peer_review (check_box, revised_sentence)
└─ grading (basic, ai_use, perspective, showcase) ← 老師勾,卡片右側
```

## 附:本次協作的觀察(供升級判斷參考)

這次設計兩個 AI 各自貢獻不同。中性陳述觀察,不下排名:

- **Codex 強的地方**:它先一步把四事件 PDF 備齊,所以對「PDF 原典裡實際存在的詞彙」有第一手立場(立場詞表的日治原詞補充)。它在實作順序上反駁我,從教學設計優先回推 schema,我接受。它在事實題(Audio Overview prompts)出了一處錯(大分 → 霧社)。
- **Claude(我)強的地方**:對既有系統實作(schema、generators、template、姓名屏蔽、深淺色雙模)熟。這次的設計張力與整合(立場詞表起手框架、評分軌道嵌位、商二乙過渡方案)由我起,Codex 修正並補強。
- **協作模式**:我起手提案 → Codex 挑戰並補實證(PDF 原典) → 我合成。**衝突點(替換 / 順序 / 視覺)是這次協作對 Lai 最有價值的部分** ─ 一個 AI 單獨給的方案會落在我或它任一方的偏誤上。

實質投資判斷:看你接下來主要在做的是「設計討論型工作」還是「執行實作型工作」。前者協作會持續加值,後者單一 AI 通常足夠。
